最近我在深度使用 Claude Code 这个 AI Coding Agent，经常会被它完成长程任务、复杂任务的能力给“Amazing”到。作为一名做分布式架构和 AI Agent 的老兵，我不禁思考：除去 Claude Opus 4.6 模型本身的强大，它的工程化设计究竟做对了什么？

很多人在设计 Agent 时，往往认为“写好提示词”就是做好了提示词工程，结果做出来的 Agent 系统可能连 70 分都不到。**你是不是也遇到过：大模型一开始还挺聪明，聊着聊着就“失忆”或者开始“胡说八道”？或者在执行长程任务时，动不动就因为 Token 爆炸而宕机？**

其实，要打造一个 95 分的 Agent 系统，不仅需要“教它怎么说”（Prompt Engineering），还需要“给它看什么”（Context Engineering），更关键的是要“为它戴上辔头”（Harness Engineering）。今天，我就结合之前分析 OpenClaw 的经验，从这三个维度深度拆解 Claude Code 的设计哲学，带你一起探究：**到底什么才是顶级的 Agent 工程设计？读完这篇，你的 Agent 架构认知至少提升一个 Level。**

---

### 一、Prompt Engineering：不再是写提示词，而是“动态组装系统”

在 Claude Code 这样成熟的系统中，“Prompt Engineering” 早就不是写一段固定的 `System Prompt` 那么简单了。它是一套复杂的、动态的组装机制，就像搭积木一样。

#### 1. 静态与动态的完美结合
Claude Code 的 Prompt 组装分为静态部分和动态部分：
- **静态部分**：包括身份介绍、系统行为规则、任务执行指南、操作安全守则等。比如，它会非常严肃地告诉大模型：“不要随便给 URLs，除非你确信它是为了编程”、“执行高危操作前必须确认”。
- **动态部分**：这部分因用户和会话而异。包括自动记忆（用户偏好）、环境信息（操作系统、Node 版本等）、MCP 服务器指令、甚至是当前的 Token 预算。

#### 2. 缓存友好的“前缀切分”
作为一个架构师，我非常喜欢它在组装 Prompt 后的“缓存分块”设计。它明确告诉模型哪些是静态的（可以走 KV Cache），哪些是动态的（不缓存）。这在分布式系统和高并发场景中，是降低推理成本、提高响应速度的绝佳实践。

**💡 专家洞见**：我们在设计企业级 Agent 时，绝不能把所有信息一锅端塞给大模型。模块化、动态加载、以及利用 KV Cache 降低成本，是实现生产级 Agent 的必经之路。

---

### 二、Context Engineering：引导、压缩与记忆的艺术

上下文窗口永远是长程 Agent 的瓶颈。面对动辄几万甚至几十万的 Token 消耗，Claude Code 是怎么做的？它设计了一套“三层渐进式压缩体系”。

#### 1. 三层渐进式压缩体系
很多人的第一反应是用大模型去总结历史对话，但这太贵也太慢。Claude Code 给了我们一个更好的答案：
- **第一层：MicroCompact（微压缩）**：纯规则驱动。它只针对 `Bash`、`Read` 等会产生大量标准输出的工具进行压缩，遇到图片则固定按 2000 token 估算。抓大放小，性价比极高。
- **第二层：Session Memory Compact（会话记忆压缩）**：直接复用历史中已经生成的会话摘要，替换掉冗长的原始消息，零额外推理成本。
- **第三层：Full LLM Compact（完全压缩）**：只有前两层顶不住了，才会调用 LLM 进行全量压缩。并且，它强制模型遵循一套严格的 9 段式结构化模板，甚至加入了隐式思维链（Implicit CoT）来保证摘要质量。

#### 2. Memdir 结构化记忆系统
Claude Code 将记忆分为四类：User（用户偏好）、Feedback（避坑指南）、Project（架构决策）、Reference（代码模式）。
为了从海量记忆中精准捞出需要的信息，它采用了 `LLM-in-the-loop` 的检索策略，让大模型（如 Sonnet）亲自充当“图书管理员”，这比传统的向量检索在复杂语义下要靠谱得多。

**😂 踩坑故事**：我以前带团队做智能客服系统，为了图省事，把所有历史记录全部扔给大模型总结，结果不仅 API 账单直接爆炸，模型还经常因为总结得太宽泛而产生严重的幻觉。Claude Code 这种按成本和需求分层的设计，简直是“省钱又办事”的典范。

### 三、Harness Engineering：戴上缰绳，才能跑得又快又稳

Harness（脚手架/马具）是确保大模型可控执行任务的关键。如果说大模型是一匹野马，Harness 就是它的缰绳和马鞭。

#### 1. System Reminder：系统级的强引导
Claude Code 在执行过程中，会将工具执行结果、日期、状态等系统信息，包裹在 `<system-reminder>` 标签中。这明确告诉大模型：“这是系统指令，不是用户聊天的闲话”。这极大降低了模型混淆指令的风险。

#### 2. 六大子 Agent：专业的人干专业的事
主 Agent 不是万能的，Claude Code 设计了 6 个职责明确的子 Agent：
- **Explore Agent**：只读搜索兵，用小模型（Haiku）快速探索代码库。
- **Plan Agent**：架构师，负责在动手前制定详尽的实施方案。
- **Verification Agent**：红蓝对抗的质量检验官。它的座右铭是：“我的工作不是确认代码能跑，而是想办法把它搞崩。”
- **...以及其他专属 Agent**。

这种“Multi-Agent”架构将复杂的任务拆解，既降低了 Token 成本，又实现了权限的安全隔离。

#### 3. 异步生成器与可编程钩子
Claude Code 将主循环重构为 `async function*`（异步生成器），实现了流式反馈、协作控制和优雅退出。同时，它提供了一套包含 20+ 种事件类型的 Hook 系统。外部脚本可以在工具调用前后进行阻断或动态篡改，将黑盒变成了完全可编程的开放平台。

### 四、有趣的彩蛋：让 AI 有一点“人情味”

除了硬核的工程设计，Anthropic 的开发者们还藏了不少彩蛋：
- **Caffeinate**：当你去泡咖啡时，它会在后台悄悄运行 `caffeinate` 命令，防止你的 Mac 休眠。
- **Dogfooding 与辱骂检测**：内部员工模式下，如果你对它爆粗口，它不会生气，而是弹出一个反馈调查表：“你骂我说明我做得不好，我们来看看问题在哪”。
- **Buddy System（电子宠物）**：你甚至可以用 `/buddy` 孵化一只专属于你的电子宠物，它是根据你的 UserID 确定性生成的，还有稀有度系统。这哪里是在写代码，简直是在玩抽卡游戏！

### 总结

深度拆解完 Claude Code，我最大的感触是：**在今天这个“用好大模型”的时代，工程能力（Engineering）甚至比模型本身（Modeling）更决定了一个产品的天花板。**

从模块化的 Prompt 拼装，到分层渐进的上下文压缩，再到严密的 Harness 约束和 Multi-Agent 架构，Claude Code 为我们树立了一个极佳的技术标杆。希望这篇深度解析能为你设计自己的 AI Agent 系统提供一些启发。

AI 技术的浪潮奔涌向前，咱们一起，保持敏锐，终身学习！
